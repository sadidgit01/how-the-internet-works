The Universal Request Lifecycle

This is the generic path almost every web request takes. Click "Show details" on any layer for the full breakdown — what it is, how it works, key details, and a real example.

1
🌐BROWSERClientClient

Initiates the HTTP request

Show details ›
What is it?

The browser (Chrome, Firefox, Safari, etc.) is the HTTP client. It's responsible for everything the user sees and interacts with.

How it works

When you type a URL or click a link, the browser parses the URL into protocol (https), hostname (example.com), path (/page), and query params. It checks its own DNS cache, local hosts file, and OS cache first before making any network call.

Key details
  1. 1Builds an HTTP/1.1, HTTP/2, or HTTP/3 request with method (GET/POST/etc.), headers (Accept, Cookie, Authorization), and body.
  2. 2Manages its own local cache: checks Cache-Control, ETag, and Last-Modified headers from previous responses.
  3. 3Enforces Same-Origin Policy (SOP) and CORS restrictions — it will block responses from foreign origins unless they explicitly allow it.
  4. 4Handles rendering pipeline: after receiving HTML, it builds DOM, then CSSOM, then Render Tree, then paints pixels.
Real example
Chrome checks localStorage, service workers, and memory cache before sending a single byte over the network.
2
🔍DNS LOOKUPDomain → IPNetwork

Translates domain name to IP address

Show details ›
What is it?

DNS (Domain Name System) is the internet's phone book. It translates human-readable domain names like example.com into machine-readable IP addresses like 93.184.216.34.

How it works

DNS resolution is recursive. If your OS cache doesn't have the answer, it asks a Recursive Resolver (usually your ISP). The resolver walks the DNS tree: Root Nameservers → TLD Nameservers (.com) → Authoritative Nameservers (example.com) → gets the IP.

Key details
  1. 1DNS lookup can add 20–120ms latency on a cold start. Browsers cache DNS records per their TTL (Time-To-Live).
  2. 2Modern apps use DNS-based load balancing — example.com resolves to different IPs in different regions (GeoDNS).
  3. 3DNSSEC adds cryptographic signatures to DNS records to prevent cache poisoning attacks.
  4. 4DNS over HTTPS (DoH) encrypts DNS queries so ISPs can't see which domains you're visiting.
Real example
example.com → Resolver → Root (.) → .com TLD → example.com authoritative → 93.184.216.34
3
🤝TCP HANDSHAKEConnection SetupNetwork

Establishes reliable connection (3-way handshake)

Show details ›
What is it?

TCP (Transmission Control Protocol) is the reliable, ordered, error-checked transport layer protocol. Before any HTTP data flows, a TCP connection must be established.

How it works

The 3-way handshake: (1) Client sends SYN packet — "I want to connect". (2) Server replies with SYN-ACK — "Okay, I'm ready". (3) Client sends ACK — "Great, let's go". Now the connection is open. This costs one full round-trip time (RTT).

Key details
  1. 1Each TCP packet has a sequence number. The receiver sends ACKs. Lost packets are automatically retransmitted.
  2. 2TCP uses flow control (receiver window) and congestion control (slow start, AIMD) to avoid overwhelming the network.
  3. 3HTTP/2 multiplexes multiple requests over a single TCP connection. HTTP/3 uses QUIC (UDP-based) to eliminate head-of-line blocking.
  4. 4TCP connection pooling (Keep-Alive) reuses open connections across multiple requests to avoid repeated handshakes.
Real example
SYN → SYN-ACK → ACK adds ~1 RTT (e.g., 50ms). HTTP/3 eliminates this with 0-RTT resumption.
4
🔐TLS HANDSHAKEHTTPS EncryptionSecurity

Negotiates encryption (adds ~1 RTT for security)

Show details ›
What is it?

TLS (Transport Layer Security) encrypts all data between client and server. Without it, anyone on the network can read your passwords and data (plain HTTP). HTTPS = HTTP over TLS.

How it works

TLS 1.3 handshake: Client sends ClientHello with supported cipher suites. Server replies with ServerHello, certificate, and key share. Client verifies certificate against trusted CA list, sends Finished. Both derive session keys. Handshake done in 1 RTT.

Key details
  1. 1The server's SSL/TLS certificate contains its public key, signed by a trusted Certificate Authority (CA) like Let's Encrypt.
  2. 2Asymmetric encryption (RSA/ECDH) is used only during handshake to exchange a shared secret. All actual data uses symmetric AES (much faster).
  3. 3TLS Session Resumption (Session IDs or Session Tickets) allows subsequent connections to skip the full handshake — 0-RTT.
  4. 4HSTS (HTTP Strict Transport Security) header forces browsers to always use HTTPS for a domain, preventing downgrade attacks.
Real example
Bank login: your password is AES-256 encrypted before leaving your browser. Even your ISP sees gibberish.
5
🛡️FIREWALL / WAFEdge SecuritySecurity

Blocks malicious traffic before it reaches your servers

Show details ›
What is it?

A Firewall filters packets based on rules (IP, port, protocol). A WAF (Web Application Firewall) operates at Layer 7 (HTTP) and understands web attacks like SQL injection and XSS.

How it works

Network Firewall: stateless (each packet independently evaluated) or stateful (tracks connection state). WAF: inspects HTTP headers, URLs, body content against attack signatures and regex patterns. Modern WAFs use ML models to detect anomalies.

Key details
  1. 1Blocks known attack patterns: SQL injection (DROP TABLE), XSS (<script>), CSRF tokens, path traversal (../../etc/passwd).
  2. 2Rate limiting at this layer: if IP sends >1000 req/min, it gets blocked or challenged with a CAPTCHA (Cloudflare, AWS WAF).
  3. 3Geo-blocking: can block all traffic from specific countries or only allow certain regions.
  4. 4DDoS protection: absorbs and scrubs volumetric attacks (millions of packets/sec) before they hit origin servers.
Real example
Cloudflare WAF blocks 57 billion threats per day before they reach origin servers.
6
☁️CDN / EDGEContent DeliveryEdge

Serves static assets from servers near the user

Show details ›
What is it?

A CDN (Content Delivery Network) is a globally distributed network of servers (Points of Presence / PoPs) that cache and serve static content from locations geographically close to users.

How it works

When a user requests example.com/logo.png: CDN edge node checks if it has a fresh cached copy. If yes → cache hit, serves immediately (5-20ms). If no → cache miss, fetches from origin, caches it, then serves. Subsequent users get the cached version.

Key details
  1. 1Cache-Control headers control CDN behavior: max-age, s-maxage, no-cache, no-store, stale-while-revalidate.
  2. 2CDN also handles SSL termination at the edge — reducing latency by ending TLS close to the user.
  3. 3Edge computing (Cloudflare Workers, Lambda@Edge) allows running JavaScript at the CDN edge for A/B testing, auth, personalization.
  4. 4CDN providers: Cloudflare, Akamai, AWS CloudFront, Fastly, BunnyCDN. They have 100s of PoPs worldwide.
Real example
Netflix uses a CDN to pre-position movie files in ISP data centers — your stream comes from a server <50ms away.
7
⚖️LOAD BALANCERTraffic DistributionRouting

Distributes requests across multiple server instances

Show details ›
What is it?

A load balancer distributes incoming HTTP requests across a pool of backend server instances. It ensures no single server becomes a bottleneck and provides high availability.

How it works

Common algorithms: Round Robin (rotate through servers evenly), Least Connections (send to server with fewest active connections), IP Hash (same client always hits same server — useful for sessions), Weighted Round Robin (powerful servers get more traffic).

Key details
  1. 1Layer 4 LB (TCP/UDP): fast, works on IP+port, no HTTP awareness. Layer 7 LB (HTTP): can route based on URL path, headers, cookies.
  2. 2Health checks: LB pings /health every few seconds. If a server fails, it's removed from the pool automatically.
  3. 3Sticky sessions: uses cookies to route same user to same server (needed for stateful apps without shared session store).
  4. 4Popular: AWS ALB/NLB, HAProxy, NGINX, Traefik, Caddy. Kubernetes uses Services + Ingress Controllers.
Real example
You have 10 app servers each handling 1000 req/s. LB gives you 10,000 req/s total capacity with automatic failover.
8
🚪API GATEWAYRequest OrchestrationRouting

Auth, rate limiting, routing, request transformation

Show details ›
What is it?

An API Gateway is the single entry point for all API calls in a microservices architecture. It abstracts the internal service topology from clients and handles cross-cutting concerns.

How it works

Request hits gateway → Authentication middleware verifies JWT/API key → Rate limiter checks quota → Request transformer normalizes data → Router determines which microservice to call → Response aggregator may combine responses from multiple services → Returns to client.

Key details
  1. 1Authentication: validates JWT tokens (checks signature, expiry, claims) or API keys before forwarding to services.
  2. 2Rate limiting: enforces per-user or per-IP quotas. Uses Redis counters (sliding window / token bucket algorithms).
  3. 3Request/response transformation: renames fields, strips internal headers, adds CORS headers, converts XML↔JSON.
  4. 4Circuit breaker pattern: if a downstream service fails repeatedly, the gateway "opens the circuit" and returns cached/default responses.
Real example
AWS API Gateway, Kong, NGINX API Gateway, Apigee, Traefik. Stripe's API gateway handles millions of requests per day.
9
🔄REVERSE PROXYNGINX / CaddyServer

Handles connections efficiently, forwards to app servers

Show details ›
What is it?

A reverse proxy sits in front of web servers and forwards client requests to them. From the client's perspective, it appears to be the origin server.

How it works

NGINX accepts thousands of concurrent connections using an event-driven, asynchronous architecture (very low memory per connection). It handles SSL termination, serves static files from disk directly, and proxies dynamic requests to app servers (Node.js, Django, etc.) via HTTP or Unix socket.

Key details
  1. 1Handles static file serving 10-100x faster than app servers — NGINX serves files directly from disk, bypassing Python/Node overhead.
  2. 2SSL/TLS termination: NGINX handles encryption so your app server only receives plain HTTP (simpler and faster).
  3. 3Compression: gzip/brotli compresses responses before sending to clients (reduces bandwidth by 60-80%).
  4. 4Connection upgrades: handles WebSocket upgrades (HTTP → WS), HTTP/2 with multiplexing, long-polling.
Real example
NGINX can handle 10,000 concurrent connections using ~2.5MB RAM. A Node.js server would need ~250MB for the same.
10
🕸️SERVICE MESHSidecar ProxyServer

Handles microservice-to-service communication (Istio/Linkerd)

Show details ›
What is it?

A service mesh manages communication between microservices inside a cluster. Each service gets a sidecar proxy (Envoy) injected alongside it that handles all network traffic.

How it works

All inter-service HTTP/gRPC traffic goes through sidecar proxies. The control plane (Istio Pilot) pushes configuration to all sidecars. This gives you observability (traces, metrics), traffic management (canary deploys, retries), and mTLS between all services — without changing application code.

Key details
  1. 1mTLS (mutual TLS): every service-to-service call is encrypted and both sides prove their identity using certificates.
  2. 2Canary deployments: route 5% of traffic to new service version, monitor error rates, gradually increase if healthy.
  3. 3Automatic retries and timeouts: if Service B is slow, the sidecar retries automatically without Service A knowing.
  4. 4Distributed tracing: each request gets a trace ID that flows through all services — you can see the full call graph in Jaeger/Zipkin.
Real example
Kubernetes + Istio: deploy 1000 microservices, all with automatic mTLS, retries, and tracing — zero extra code.
11
⚙️APP SERVERBusiness LogicServer

Executes your application code and processes the request

Show details ›
What is it?

The application server is where your actual code runs. It processes the business logic: validates input, calls databases, performs calculations, enforces business rules, and constructs the response.

How it works

Request arrives (HTTP) → Framework router matches URL to handler → Middleware chain (auth, logging, body parsing) → Controller/Handler runs business logic → Calls DB/Cache/external APIs → Serializes response (JSON/HTML) → Returns HTTP response.

Key details
  1. 1Frameworks: Express.js (Node), Django/FastAPI (Python), Spring Boot (Java), Laravel (PHP), Rails (Ruby), Gin (Go).
  2. 2Runs in multiple worker processes or threads (or async event loop for Node/FastAPI) to handle concurrent requests.
  3. 3Stateless by design — no session data stored locally. Auth state in JWTs or external session stores (Redis).
  4. 4Health check endpoints (/health, /ready) let the load balancer know if the instance is alive and ready for traffic.
Real example
FastAPI: request → Pydantic validation → SQLAlchemy DB query → Pydantic serialization → JSON response in <10ms.
12
⚡CACHE LAYERRedis / MemcachedData

In-memory data store for fast reads (microseconds vs milliseconds)

Show details ›
What is it?

A cache is an in-memory key-value store that holds frequently accessed data. Reading from cache takes ~0.1ms vs ~10ms for a database query — 100x faster. Redis is by far the most popular.

How it works

Cache-aside pattern: app checks cache first. Cache hit → return cached value. Cache miss → query DB, store result in cache with TTL, return value. Write-through: write to cache and DB simultaneously. Write-behind: write to cache, async write to DB later.

Key details
  1. 1TTL (Time-To-Live): cached values automatically expire after N seconds, preventing stale data from being served forever.
  2. 2Cache eviction policies: LRU (Least Recently Used) — when cache is full, remove the entry accessed longest ago.
  3. 3Redis supports rich data types: Strings, Lists, Sets, Sorted Sets, Hashes, Streams — not just simple key-value.
  4. 4Redis Pub/Sub: can also be used as a message broker for real-time features like live notifications and presence.
Real example
Twitter caches your home timeline in Redis. Without cache, rendering your feed would require ~100 DB queries per load.
13
🗄️DATABASEPostgreSQL / MySQL / MongoDBData

Persistent, reliable storage for all application data

Show details ›
What is it?

The database is the source of truth for all persistent data. Unlike cache, data here survives server restarts. Relational databases (PostgreSQL, MySQL) store data in tables with relationships. NoSQL (MongoDB) stores documents.

How it works

App server sends SQL query (or ORM-generated query) → DB parses and optimizes query plan → Executor reads/writes from storage engine (B-tree/LSM-tree indexes) → Returns result set → App server maps rows to objects.

Key details
  1. 1Indexes: B-tree index on a column makes lookup O(log n) instead of O(n) full table scan. Critical for performance at scale.
  2. 2ACID transactions: Atomicity, Consistency, Isolation, Durability. Your bank transfer either fully succeeds or fully rolls back.
  3. 3Read replicas: write to primary, read from replicas. Scales read capacity horizontally. Replication lag can be 0-100ms.
  4. 4Connection pooling (PgBouncer): DB connections are expensive. Pool reuses a small number of connections across many app requests.
Real example
PostgreSQL with proper indexes handles 10,000+ queries/sec on a single instance. Without indexes, 100 queries/sec.
14
📬MESSAGE QUEUEKafka / RabbitMQ / SQSAsync

Decouples services; handles async background tasks

Show details ›
What is it?

A message queue is a buffer between services. Instead of doing slow work synchronously (sending email, processing image, generating report), the app publishes a message to the queue and immediately returns to the user. Workers consume and process messages asynchronously.

How it works

Producer (App Server) publishes message to queue/topic → Queue durably stores messages → Consumer (Worker) polls and processes messages → Consumer ACKs message on success (removes from queue) or NACKs on failure (requeues for retry).

Key details
  1. 1At-least-once delivery: Kafka/SQS guarantee each message is delivered at least once. Your consumer must be idempotent (handling duplicate processing safely).
  2. 2Dead Letter Queue (DLQ): after N failed retries, messages go to DLQ for manual inspection and debugging.
  3. 3Kafka partitions allow parallel consumption: 10 partitions = 10 consumers can process simultaneously, giving horizontal scale.
  4. 4Use cases: email/SMS sending, image resizing, PDF generation, payment processing, ML model inference, audit log writes.
Real example
Uber: when you book a ride, the API returns instantly. Driver matching, notifications, surge pricing all happen via queues.
15
👷BACKGROUND WORKERSCelery / BullMQ / SidekiqAsync

Consumes queue messages and executes async jobs

Show details ›
What is it?

Background workers are separate processes that consume tasks from the message queue. They run independently from the web servers, enabling CPU-intensive or slow operations without blocking HTTP responses.

How it works

Worker process starts → connects to queue (Redis/RabbitMQ/SQS) → polls for jobs → deserializes job data → executes task function (send email, resize image, etc.) → marks job complete → polls again. Multiple workers run in parallel.

Key details
  1. 1Celery (Python) + Redis/RabbitMQ: most popular combo for Python apps. BullMQ for Node.js, Sidekiq for Ruby.
  2. 2Job priorities: urgent jobs (password reset email) skip the queue via priority lanes over bulk jobs (weekly newsletter).
  3. 3Scheduled jobs (cron-like): Celery Beat, pg-boss, Agenda — run tasks on a schedule (generate daily reports at 2am).
  4. 4Monitoring: Flower (Celery), Bull Board (BullMQ) — dashboards showing queued/active/failed jobs in real-time.
Real example
When you sign up on any website, the HTTP response returns in 200ms. The welcome email arrives seconds later — sent by a worker.
16
🔌EXTERNAL APISThird-party ServicesAsync

Payment gateways, email, SMS, OAuth, AI services

Show details ›
What is it?

Most apps integrate with third-party APIs: Stripe for payments, SendGrid for email, Twilio for SMS, AWS S3 for file storage, Google OAuth for login, OpenAI for AI features. Your app server calls these as an HTTP client.

How it works

App server makes outbound HTTP request to external API with API key in Authorization header → External service processes request → Returns JSON response → App server handles success/error → Often these calls happen in background workers to avoid blocking the main request.

Key details
  1. 1Circuit breaker pattern: if Stripe API is down, stop sending requests (circuit open), use cached/fallback response, retry after timeout.
  2. 2Webhook callbacks: instead of polling, register a URL with the third party. They POST to your URL when an event happens (payment succeeded).
  3. 3Rate limits: external APIs have per-minute/day limits. Implement exponential backoff + jitter for retries.
  4. 4Secret management: API keys stored in environment variables or secret stores (AWS Secrets Manager, HashiCorp Vault) — never in code.
Real example
Checkout flow: App → Stripe API (charge card) → Stripe webhook → App (update order status) → SendGrid (send receipt email).
17
↩️RESPONSE PATHBack to BrowserResponse

Response travels back through the same stack in reverse

Show details ›
What is it?

The HTTP response (status code + headers + body) travels back through the same layers in reverse order. Each layer may add headers, compress data, or apply transformations before passing it back to the client.

How it works

App Server builds response → NGINX compresses (gzip) and adds headers → API Gateway adds CORS headers → Load Balancer forwards → CDN may cache the response for next users → TLS encrypts → TCP segments delivery → Browser receives, decompresses, parses, renders.

Key details
  1. 1HTTP status codes: 200 OK, 201 Created, 301/302 Redirect, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 500 Internal Server Error.
  2. 2Response headers set browser behavior: Cache-Control, Set-Cookie, Content-Type, Content-Encoding (gzip), CORS headers (Access-Control-Allow-Origin).
  3. 3TTFB (Time To First Byte): how long from request send to first byte of response received. Key performance metric.
  4. 4Browser rendering: HTML → DOM, CSS → CSSOM, combined → Render Tree → Layout → Paint → Composite. First Contentful Paint (FCP) is the visible milestone.
Real example
curl -I example.com shows all response headers. Chrome DevTools Network tab shows TTFB, download time, and waterfall for every asset.

The Netflix Example — "Searching for a Movie"

Let's make this concrete. You open Netflix and search for "Stranger Things" from your laptop in a coffee shop. Here's the exact same journey, step by step.

1
🌐Browser ClientClient

You type "Stranger Things" in the Netflix search bar and press Enter. The browser builds an HTTPS request. It's ready — but it has a name, not an address.

request
GET https://www.netflix.com/search?q=stranger+things
2
🔍DNS LookupNetwork

The browser asks "Where is netflix.com?" The DNS system responds: 54.237.154.123. Now the browser knows the physical server location.

3
🤝TCP Handshake (SYN → SYN/ACK → ACK)Network

Browser and Netflix's server do the three-way handshake. A reliable connection is established.

4
🔐TLS HandshakeSecurity

Browser and server negotiate encryption. Certificates are exchanged. The browser confirms: "Yes, this is really Netflix." From now on, all data is encrypted — no one at the coffee shop can see what you're searching for.

5
🛡️Firewall / WAFSecurity

The encrypted request hits Netflix's network. The firewall checks: "Is this a known attacker? Is the data well-formed?" It passes. You're not a bot.

6
☁️CDN / EdgeEdge

The request arrives at a CDN server near your city. The CDN checks: "Do I have search results for 'Stranger Things' cached?" Probably not — search results are personalized. It forwards the request to Netflix's main data center.

7
⚖️Load BalancerRouting

The request arrives at Netflix's data center. The load balancer looks at its 500 app servers and picks one: "Server 342 has the fewest active connections. Send it there."

8
🚪API GatewayRouting

The request hits the API Gateway. The gateway reads the path /search and knows: "This goes to the Search Service." It also checks your session cookie to confirm you're logged in. You are. Forwarded.

9
🔄Reverse Proxy (NGINX)Server

NGINX receives the request. "Is this search result cached?" No — personalized data isn't cached here. "Let me decompress it and pass it to the Node.js app server."

10
🕸️Service Mesh SidecarServer

The sidecar proxy notes: "Request entering the Search Service. Let me track its latency and make sure the service is healthy."

11
⚙️App Server (Node.js)Server

Your Node.js code finally gets the request:

search.route.js
app.get('/search', async (req, res) => {
    const query = req.query.q;  // "stranger things"
    const userId = req.user.id; // your user ID

    // Step 12: Check cache first
    const cached = await redis.get(`search:${userId}:${query}`);
    if (cached) return res.json(JSON.parse(cached));

    // Step 13: Cache miss — hit the database
    const results = await db.query(`
        SELECT * FROM movies 
        WHERE title ILIKE '%stranger things%'
        ORDER BY popularity DESC
        LIMIT 20
    `);

    // Cache it for next time
    await redis.set(`search:${userId}:${query}`, JSON.stringify(results), 'EX', 300);

    // Step 14 (async): Publish to message queue
    // "Log this search for recommendations later"
    await queue.publish('search.log', { userId, query, timestamp: Date.now() });

    // Send the response
    res.json(results);
});
12
⚡Cache Layer (Redis)Data

"Search cache for 'stranger things'? Not found. Moving on."

13
🗄️Database (PostgreSQL)Data

The database runs the query. It finds 3 matching results: "Stranger Things" (seasons 1–4), "Stranger Things: The Game", and a documentary "The Strange World Of...". It returns the data in ~20ms.

14
📬Message Queue (Kafka)Async

After the response is sent, the Node.js app publishes a message: "User 12345 searched for 'stranger things'." This message waits in the queue.

15
👷Background WorkersAsync

A separate worker process picks up that message and:

  • Updates the trending searches dashboard
  • Refines your recommendation algorithm
  • Logs the search for analytics

All of this happens after you get your response — you never wait for it.

16
🔌External APIs (maybe)Async

If Netflix uses a third-party content moderation API to filter search results for age-appropriate content, it calls that here. Or if it uses an AI service to suggest related searches. Not always needed, but available.

17
↩️Response Path (continued)Response

The response travels back:

  • Service mesh: "Response leaving the Search Service. Latency recorded: 45ms."
  • Reverse proxy: NGINX compresses the JSON response with gzip (reduces size by 70%), caches it briefly for identical requests, and sends it back.
  • API Gateway: The gateway adds some security headers and passes the response through.
  • Load Balancer: Routes the response to the correct client connection.
  • CDN / Edge: "I'll cache this search result for 5 minutes in case nearby users search the same thing."
  • Firewall: Passes through without issue.
  • TLS: Encrypted the whole way back.
  • TCP: Arrives in order.
  • Browser: Receives the JSON, renders the search results. You see the "Stranger Things" poster.
Total round trip, click to poster on screen ~200–400ms
